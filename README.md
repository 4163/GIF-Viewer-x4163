## Helper Predecode Implementation

The main code path is in:

```text
source/GIFFile.cs
```

Behavior:

- `GifFile.LoadFromPath()` first checks for the Quivi/Pillow helper runtime next to the built executable.
- If present, GIF Viewer asks the helper to decode frames into temporary BMP files plus `delays.txt`.
- GIF Viewer loads those BMPs into 32bpp in-memory `Bitmap` frames.
- Playback then copies pixels from those predecoded frames.
- If the helper is missing or fails, GIF Viewer falls back to its original internal decoder.

Helper lookup order:

1. `webp_extract_pillow_runtime/webp_extract_pillow/webp_extract_pillow.exe`
2. `webp_extract_pillow.exe`

## Performance Notes

The helper path shifts work from playback time to load time.

Potential benefits:

- Less decoder work during playback.
- More predictable frame switching after load.

Costs:

- Higher initial load time.
- Higher RAM use, because full frames are materialized.
- Large real-world GIFs still need broader validation.

Future scaling work should aim to fix and optimize the resampling behavior used by the scaling methods. The helper predecode path improves playback, but expensive scaling can still dominate frame presentation for large images. Future work should keep the quality selector functional while reducing the cost of high-quality resizing.

## Public Repository Hygiene

Before publishing the repository:

- Keep `source/`, `diffs/`, `backups/`, and `tools/` if sharing development history.
- Keep the root runnable build if sharing a ready-to-run copy.
- Do not include private test GIFs.
- Keep helper runtime files only if distributing a runnable modified build.
- Review licenses for GIF Viewer and the helper runtime dependencies.

No private test paths or personal sample names should remain in the notes/scripts.
